# refmesh 要件定義・仕様書 (Hybrid Semantic Graph CLI)

## 1. プロジェクトの目的
コーディングエージェント（Claude Code等）が外部の公式ドキュメントやリファレンスから得た情報を、自律的に「概念（ノード）」と「関係性（エッジ）」のネットワークとして抽出し、ローカル環境に保存・検索できるようにする。自然言語による曖昧な検索（ベクトル検索）と、体系的なネットワーク探索（グラフ検索）を組み合わせることで、情報の取りこぼしを防ぎ、エージェントに高度な長期記憶（意味記憶）を提供する。

## 2. システムの全体像とアーキテクチャ
* **動作形態:** TypeScript + Node.js製のコマンドラインツール（CLI）。
* **ハイブリッド・データストア:**
  1. **Vector Store (LanceDB):** ノードの意味（テキスト）をベクトル化して保持し、検索のエントリポイント（起点）を特定する。
  2. **Graph Store (Kùzu):** 特定されたノードを起点に、N階層の知識ネットワーク（依存関係、構成要素等）を保持・抽出する。

## 3. 検索機能の高度化 (`search` コマンド)

単一のキーワードだけでなく、自然言語による文章クエリを受け入れ、関連する知識の束を網羅的に引き出す。

### 3.1. 複数候補の同時抽出プロセス
検索実行時、内部で以下のハイブリッドステップを実行する。

1. **セマンティック・リトリーバル (Vector Layer):**
   * 入力された自然言語クエリをベクトル化し、Vector Storeから類似度の高いノードを **最大 $N$ 個（デフォルト: 5）** 抽出する。
   * 指定されたしきい値（Threshold）を下回るノードはノイズとして除外する。
2. **マルチ・起点・トラバーサル (Graph Layer):**
   * 抽出されたすべての候補ノードに対し、それぞれを起点としたグラフ探索（`--depth` に基づく）を並列または連続で実行する。
3. **データ統合と重複排除 (Merge & De-duplicate):**
   * 複数の起点から得られたノードおよびエッジの集合を一つのグラフ構造に統合し、重複を排除したユニークなコンテキストとして出力する。

### 3.2. コマンドオプション
* `refmesh search "<query>"`
* `--depth <n>`: グラフ探索の深さ（デフォルト: 1）
* `--limit <n>`: ベクトル検索で取得する候補ノードの最大数（デフォルト: 5）
* `--threshold <value>`: 類似度の最小しきい値（0.0 〜 1.0）

## 4. 登録機能の高度化 (`register` コマンド)

エージェントが抽出した知識データ（JSON形式）を受け取り、両データベースへの書き込みと同期を行う。

### 4.1. 処理フロー
1. **入力バリデーション:** エッジの種類等が定義済みEnumに準拠しているか確認。
2. **Embeddingの生成:** ノードの `id`、`description`、`details` を結合し、ベクトル表現を生成。
3. **永続化:**
   * Vector Storeに「ベクトルとIDのペア」を保存。
   * Graph Storeに「ノードとエッジの構造」をUpsert（MERGE処理）。

### 4.2. 重複検知と自動リンク
* 新しい概念を登録する際、まずVector Storeで類似度チェックを行う。
* 極めて高い類似度を持つ既存ノードが発見された場合、エージェントに対し「既存の `Node ID` と同一である可能性」を警告し、IDの正規化や `SAME_AS` エッジによる接続を促す。

## 5. ベクトル化（Embedding）の技術要件

* **コア技術:** `sentence-transformers` アーキテクチャを採用する。
* **採用モデル:** 日本語・英語の公式ドキュメントを横断的に高精度で意味検索できるよう、多言語対応のモデル（例: `paraphrase-multilingual-MiniLM-L12-v2` 等）を標準とする。
* **TypeScript (Node.js) 統合:**
  * ONNXフォーマットに変換したモデルを `@xenova/transformers` (Transformers.js) でロードし、Node.jsプロセス内でネイティブにベクトル生成を実行する。これにより、Python環境への依存を排除しCLIとしてのポータビリティを確保する。

## 6. データモデル（グラフスキーマ定義）

### 6.1. ノード（Concept）
* `id` (string, PK): 一意の名称。
* `description` (string): 概要。ベクトル化の主要対象。
* `details` (string): 補足情報やコードスニペット。

### 6.2. エッジ（Relationships: Enum定義）
| カテゴリ | エッジタイプ | 意味・役割 |
| :--- | :--- | :--- |
| **構造・分類** | `IS_A`, `PART_OF`, `CONTAINS` | 概念の親子・内包関係 |
| **依存・実装** | `DEPENDS_ON`, `IMPLEMENTS`, `EXTENDS` | 動作条件や仕様の継承 |
| **データフロー**| `CONSUMES`, `PRODUCES`, `MUTATES` | データの入出力と状態変化 |
| **比較・関連** | `ALTERNATIVE_TO`, `INTEGRATES_WITH`, `RELATED_TO` | 代替手段や連携可能な周辺技術 |
| **同一性解決** | `SAME_AS` | 表記揺れにより分離したノードの事後的な結合 |
| **ライフサイクル**| `REPLACES`, `DEPRECATES` | 技術の変遷 |

## 7. エージェントスキル定義（運用ルール）

エージェントのシステムプロンプトに組み込む運用ガイドライン。

1. **目的ベースの検索:** 単語の完全一致を気にする必要はない。タスクの目的（例：「Reactでの安全な状態管理」）をそのままクエリとして `search` し、関連する複数の技術要素を一度に引き出すこと。
2. **カノニカルネームの採用:** 登録時は可能な限り公式で一般的な名称を `id` とすること。
3. **網（Mesh）の構築:** 既存ノードとの関連性に気づいた場合は、新規ノードを作るだけでなく、必ず適切なエッジを用いて知識をリンクさせること。
