# 自律型エージェント向け ナレッジグラフ構築CLI（`refmesh`）要件定義書

## 1. プロジェクトの目的
コーディングエージェント（Claude Code等）が外部の公式ドキュメントやリファレンスから得た情報を、自律的に「概念（ノード）」と「関係性（エッジ）」のネットワークとして抽出し、ローカルのグラフデータベースに保存・検索できるようにする。これにより、エージェントにプロジェクト横断的な長期記憶（意味記憶）を持たせ、高度な推論や意思決定を支援する。

## 2. システムの全体像とアプローチ
* **動作形態:** TypeScript + Node.js製のコマンドラインツール（CLI）。
* **データストア:** Kùzu（プロセス組み込み型のローカルグラフデータベース）。
* **アーキテクチャ:** 階層に縛られないフラットな「ノードとエッジのリスト」をJSONで受け渡し、N階層の深い知識構造を表現する。

## 3. 機能要件

### 3.1. 知識の登録機能 (`register`)
エージェントが抽出した知識データ（JSON形式）を受け取り、グラフDBへ書き込む。
* **入力方式:** 標準入力（パイプ `|`）、またはファイルパス指定（`-f <path>`）。
* **処理内容:**
  * 入力JSONに対する厳密なスキーマバリデーションを実行。
  * 既存ノードとの重複を防ぐため、ノードの `id` に基づくUpsert（MERGE）処理を実行。

### 3.2. 知識の自動統合と参照チェック（Identity-based Integration）
* すべての登録処理はノードの `id`（名称）を基準に行われる。異なるドキュメントから抽出されたデータでも、同一 `id` を持つノードはグラフ内で自動的に集約・共有される。
* `relationships` において、`concepts` リストに含まれない `id` が指定された場合、CLIは既存のDB内を探索し、存在すればエッジを張る。存在しない場合はバリデーションエラーとして処理を中断する。

### 3.3. 知識の検索機能 (`search`)
エージェントが必要な知識を引き出すためのインターフェース。
* **入力方式:** `refmesh search <キーワード> [--depth <数値>]`
* **オプション:** `--depth`（デフォルト: 1）。指定された深さまでの関連ノード（子孫・周辺知識）を再帰的に取得する。
* **出力方式:** エージェントが文脈として理解しやすい構造化テキスト、またはJSON。

### 3.4. スキーマ・定義の出力機能 (`types`)
エージェントが自身の挙動を調整したり、利用可能な関係性を把握したりするためのコマンド。
* **入力方式:** `refmesh types`
* **出力方式:** 利用可能なエッジのEnumリストとその説明、要求されるJSONスキーマを標準出力に返す。

## 4. データモデル要件（グラフスキーマ定義）

### ノード（エンティティ）
1. **Reference（情報源）:** `url` (PK), `title`
2. **Concept（知識・概念）:** `id` (PK, 名称), `description`, `details` (オプショナルな補足情報やコードスニペット)

### エッジ（リレーションシップ: Enum定義）
CLI内部で以下のリストを正として管理し、登録時のバリデーションに使用する。

| カテゴリ | エッジタイプ | 意味・役割 |
| :--- | :--- | :--- |
| **構造・分類** | `IS_A` | AはBの一種である |
| | `PART_OF` | AはBの一部・構成要素である |
| | `CONTAINS` | AはBを内包している |
| **依存・実装** | `DEPENDS_ON` | Aが機能するにはBが必要である |
| | `IMPLEMENTS` | AはB（仕様等）を実装している |
| | `EXTENDS` | AはBの機能を拡張・継承している |
| **データフロー** | `CONSUMES` | AはB（データ等）を受け取る |
| | `PRODUCES` | AはBを生成・出力する |
| | `MUTATES` | AはBを変更・破壊する |
| **比較・関連** | `ALTERNATIVE_TO` | AはBの代替手段である |
| | `INTEGRATES_WITH` | AはBと連携・接続できる |
| | `RELATED_TO` | AとBは関連している |
| **ライフサイクル** | `REPLACES` | AはBを置き換える新しい技術である |
| | `DEPRECATES` | AはBを非推奨としている |

*(※ `DESCRIBES` エッジは `Reference` → `Concept` の紐づけ専用としてシステムが内部で自動利用する)*

## 5. 入出力 JSON インターフェース（スキーマ定義）

エージェントが `register` コマンドに渡すデータ構造。

```json
{
  "reference": {
    "url": "[https://react.dev/reference/react/hooks](https://react.dev/reference/react/hooks)",
    "title": "Built-in React Hooks"
  },
  "concepts": [
    {
      "id": "useState",
      "description": "コンポーネントに状態変数を追加するためのHook",
      "details": "const [state, setState] = useState(initialState);"
    },
    {
      "id": "React Hooks",
      "description": "Reactの状態管理やライフサイクル機能へのフック群"
    }
  ],
  "relationships": [
    {
      "source": "useState",
      "target": "React Hooks",
      "type": "PART_OF",
      "reason": "useStateはReactが提供する標準Hookの一つであるため"
    }
  ]
}
```

## 6. エージェントスキル定義（Agent Skill / Custom Instructions）

エージェントが本ツールを自律的に運用するためのルール。エージェントのシステムプロンプト等に組み込む。

### 6.1. トリガー条件
* 未知のライブラリやAPIのドキュメントを読み込んだとき。
* 人間から「知識を整理して」と明示的に指示されたとき。

### 6.2. ワークフロー（Action Flow）
1. **事前検索 (Search Before Register):**
   * 抽出した概念がすでに存在しないか `refmesh search` で確認し、既存の `id` との不一致や表記ゆれを防ぐ。
2. **構造化 (Extract & Structure):**
   * `refmesh types` で定義されたエッジタイプのみを使用し、フラットなノード・エッジ形式のJSONを構築する。
3. **登録と自己修正 (Execute & Self-Correction):**
   * `refmesh register` を実行。バリデーションエラーが発生した場合は、エラーメッセージに基づきJSONを修正して再試行する。

### 6.3. 知識の活用
* 実装判断やデバッグの際、過去に蓄積した知識が必要な場合は `refmesh search` を実行し、その結果をコンテキストとして思考プロセスに含めること。
